### Answer 1

**正答:** **D**

**正解（D）の理由**

`Transaction::logTransaction` は宣言上 **`= 0` により純粋仮想**です。そのため `Transaction::Transaction()` が `init()` 経由で **`logTransaction()` を仮想呼び出し**したとき、この時点での動的类型は基底の構築中なので **`Transaction` 側の vtable が使われる**一方、純粋仮想のために **処理系依存の異常処理（多くは `pure virtual method called` 等を出して異常終了）** が起きやすくなります。クラス外に `Transaction::logTransaction` の定義があっても、**コンストラクタ中は普通の動的束縛でその本体に載り替わるとは限らない**（処理系により「純粋仮想呼び出し」扱い）ため、問題の構成では実行時異常終了が正答です。また本ソースはコンストラクタ本文に `BuyTransaction::logTransaction` が呼ばれる箇所はなく、異常終了後に `[BuyTransaction]` が出て完了する経路にもなりません。

**誤答の理由**

- **A** … クラス外定義があるので「基底実装まで通常の仮想呼び出しで到達して `[Transaction]` が出る」と読みやすいですが、基底コンストラクタ実行中は動的类型がまだ基底で、しかも **`logTransaction` は基底で `= 0` のまま解釈されている**ので、処理系により **異常終了**し、この設問の意図する正しい結果（D）とは一致しません。  
- **B** … 基底コンストラクタ中は **`BuyTransaction::logTransaction` に動的束縛されない**ので（第9項の典型）、B は誤りです。  
- **C** … `logTransaction()` は `Transaction::init()` で **せいぜい一度**評価されようとしており、その途中で異常終了するため両方並んで表示される、という状況にはなりません。

**実行される順序（`BuyTransaction b;` 1 回分）**

異常終了するまでに到達しうる順序です（ソース上は設問 Q1 のコードに対応）。

1. `main` の `BuyTransaction b;` により、`BuyTransaction` の構築が開始される。  
2. **基底** `Transaction` のコンストラクタへ制御が入る（`Transaction()`）。  
3. `Transaction()` 本文で `init();` が呼ばれる。  
4. `Transaction::init()` に入り、本文で `logTransaction();` が **仮想呼び出し**として評価される。  
5. 処理系により **純粋仮想呼び出しとして異常終了**（コンソールへのメッセージや `terminate` は環境依存）し、基底も派生も最後まで構築されない。そのため、`[Transaction]` / `[BuyTransaction]` のような **問題文どおりの正常なログ出力に至らない**経路となる。

### Answer 2

**正答:** **A**

**正解（A）の理由**

C++ では、派生クラス `BuyTransaction` のオブジェクトを作るとき、**先に基底サブオブジェクト `Transaction` のコンストラクタが実行**されます。基底のコンストラクタが走っている間は、動的型はまだ `Transaction` とみなされるため、`init()` 内の `logTransaction()` は仮想呼び出しでも **`Transaction::logTransaction` に解決**されます。したがって `[Transaction] Transaction` だけが出力されます。`logTransaction` にはクラス外定義があり純粋仮想ではないので、この呼び出しは規格上明確に定義された挙動です。

**誤答の理由**

- **B** … 「`virtual` だから常に最派生クラスへ飛ぶ」と考えると選びがちですが、**基底コンストラクタ実行中は派生側のオーバーライドは使われません**。  
- **C** … `logTransaction()` は `Transaction::init()` から **1 回だけ**呼ばれます。`BuyTransaction` のコンストラクタ本文では `logTransaction` を呼んでいないため、**2 行は出ません**。  
- **D** … 問題文の選択肢どおり実行時異常終了となるのは「純粋仮想をコンストラクタ経由で仮想呼び出しする」側（Q1）のパターンです。本問（Q2）では `Transaction::logTransaction` に本体があり基底で `= 0` でないので、**コンストラクタ中は基底版に解決される**のみで異常終了にはなりません。

**実行される順序（`BuyTransaction b;` 1 回分）**

コンストラクタ本体・補助関数の「どの文がいつ」の流れです（ソース上の概略位置は設問コードに対応）。

1. `main` の `BuyTransaction b;` により、`BuyTransaction` の構築が開始される。  
2. **基底** `Transaction` のコンストラクタへ制御が入る（`Transaction()`）。  
3. `Transaction()` 本文で `init();` が呼ばれる。  
4. `Transaction::init()` に入り、本文で `logTransaction();` が呼ばれる。  
5. `Transaction::logTransaction()` が実行され、`std::cout` で `[Transaction] Transaction` が出力される。  
6. `init`、`Transaction()` から順に戻り、基底サブオブジェクトの構築が完了する。  
7. **派生** `BuyTransaction` のコンストラクタ（暗黙のデフォルト）が実行されるが、本文でログ出力はしない。  
8. `BuyTransaction b;` が完了し、`main` の次の処理へ進む（本コードには無い）。

（補足）メンバの初期化子があればコンストラクタ本文より前に走りますが、本コードの両クラスにメンバ変数がないので、上記の順序がそのまま成立します。

### Answer 3

**正答:** **C**

- 名前を `init` に変えても、仮想関数をコンストラクタから呼ぶ構図はそのままです。  
- `const` かどうかも本質ではありません。  
- 「慣習名だから安全」ということもありません。

### Answer 4

**正答:** **B**

- B が本文の改善案に最も忠実です。  
- ポイントは、**派生クラス側で必要なログ情報を先に作り、それを基底クラスへ渡し、基底クラスでは非仮想関数で処理する**ことです。  
- C は引数付きになっていても、`logTransaction` を仮想関数に戻しているので NG です。  
- D は「改善案っぽく見える」強いひっかけですが、仮想関数依存を増やしており逆方向です。

### Answer 5

**正答:** **B**

- `static` の狙いは、**まだ完全に構築されていないオブジェクトの状態に依存しない**ことです。  
- A は仮想関数と `static` を混同した誤答です。  
- C, D も `static` の性質ではありません。

### Answer 6

**正答:** **C**

- 第9項の主眼は、「危険な呼び出しをうまく成立させるテクニック」ではなく、**そもそもその依存を設計から外す**ことです。  
- `explicit` や `private` は補助的な記述であり、本質的な安全性の根拠ではありません。
